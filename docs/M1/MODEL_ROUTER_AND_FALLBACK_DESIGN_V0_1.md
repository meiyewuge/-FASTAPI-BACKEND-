## 一、定位与核心原则

### 1.1 四层架构定位

| 层级 | 名称 | 职责 | 事实权限 |
|---|---|---|---|
| 第一层 | Brief 理解层 | 理解员工要写什么、给谁看、在哪个平台发 | 无事实产出 |
| 第二层 | 9080 召回层 | 从事实库、合规库、风格库、东方状态美学引擎拿材料 | 唯一事实入口 |
| 第三层 | 大模型生成层 | 根据 Brief + used_materials + 平台工艺卡生成内容 | 零事实产出 |
| 第四层 | G1-G6 六硬门 | 合规/事实/平台结构/品牌一致/格式完整质检 | 事实校验出口 |

> **核心公式：库负责"真"，模型负责"会写"，合规门负责"不能乱"，人负责"值不值得发"。**

### 1.2 大模型权限边界

**大模型可以兜底（7 项）：**

1. Brief 理解
2. 内容结构生成
3. 语气改写
4. 平台适配
5. 东方状态美学表达
6. 多版本扩写
7. 草稿优化

**大模型不得兜底（8 项）：**

1. 产品事实
2. 检测数据
3. 功效依据
4. 法规依据
5. 用户案例
6. 医美建议
7. 经营数据
8. approved 素材缺失

> **铁律**：事实必须来自 9080 的 approved 素材和 used_materials。资产型内容必须全部来自 9080 approved 素材，平台灵感只能做表达参考，不能做事实依据。（来源：AI 写稿助手 V1.3）

## 二、模型路由设计

### 2.1 四角色模型路由

| 角色 | 职责 | 选用标准 | 典型场景 |
|---|---|---|---|
| primary_model | 主模型，负责主要出稿 | 稳定、可控、成本清楚 | 事实严谨型任务：品牌科普/IP观点/门店诊断 |
| fallback_model | 兜底模型，主模型失败时接管 | 高可用、低延迟 | 主模型超时/质量不达标/风格偏移 |
| review_model | 审稿模型，自检+找风险+看结构 | 高精度、擅长发现错误 | G1 前置自检/结构完整性/合规预扫描 |
| rewrite_model | 改写模型，平台改写+口语化+美感润色 | 低成本、高并发、中文审美强 | 小红书/抖音/视频号多版本改写 |

### 2.2 任务类型 → 模型路由映射

| 任务类型 | 主路由 | 特殊约束 |
|---|---|---|
| 事实严谨型（品牌科普/检测引用/功效说明） | primary_model（稳定可控） | 必须带 used_materials，模型不得补充事实 |
| 状态美学表达（东方状态美学版本） | primary_model + rewrite_model 润色 | 允许词/谨慎词/禁用词受合规门约束 |
| 小红书/抖音/视频号改写 | rewrite_model（低成本批量多版本） | 各平台禁区词受 G1 四出口差异约束 |
| 高风险内容（敏感肌/医美/术后/招商） | primary_model + review_model 双检 | 不依赖免费模型直接过稿，必须 G1+人审 |
| IP 观点稿扩展 | rewrite_model 扩写 + primary_model 结构 | Kimi 可作为候选，但受限扩写模式 |
| 长文扩写 | Kimi 候选 | 只吃 used_materials 扩写，不补事实 |

### 2.3 模型调用流程

```
Brief 输入
    ↓
Brief 理解层（primary_model 理解意图）
    ↓
9080 召回层（事实/合规/风格/引擎素材）
    ↓
used_materials 组装
    ↓
大模型生成层（primary_model 出稿）
    ↓
[质量检查] → 不达标 → fallback_model 重试（≤2 次）
    ↓
[平台改写] → rewrite_model 生成 N 版
    ↓
[自检预扫] → review_model 结构+合规预检
    ↓
G1-G6 六硬门
    ↓
candidate_review（signoff_approved / rejected）
    ↓
人工签发
```

## 三、Kimi 受限扩写模式

### 3.1 Kimi 定位

Kimi 可作为候选扩写模型之一，适合：

- 长文扩写
- 中文表达润色
- 状态美学版本
- IP 观点稿扩展
- 把专业内容写得更舒服、更有画面感

### 3.2 Kimi 不得作为

- **事实源**（只能吃 used_materials 扩写，不能自己补事实）
- **唯一生产底座**（免费额度/接口稳定性/并发/隐私/可用性都可能变化）

### 3.3 Kimi 受限扩写 10 条件（与 G1 合规门映射表一致）

1. 三清单补丁完成 ✅
2. 吴哥确认生效 ⏳ 待拍板
3. Kimi 只扩写 draft_candidate
4. Kimi 不得新增事实
5. Kimi 不得越过 used_materials
6. Kimi 输出必须过 G1 合规门
7. Kimi 输出不得直接进入 approved（→ signoff_approved）
8. Kimi 输出不得自动发布
9. 高风险内容必须进入 must_sign
10. Kimi 只做表达扩写，不做事实定稿

## 四、可插拔模型路由机制

### 4.1 设计原则

- 不押宝单一模型，尤其是免费模型
- 模型路由做成可插拔架构，随时可换
- 以后可切换 DeepSeek/通义/豆包/Kimi/硅基流动模型，甚至多模型混用
- 模型选择由任务类型+成本约束+质量要求决定，不人为绑定

## 五、模型失败 Fallback 机制

### 5.1 触发条件

| 触发 | 检测方式 | 动作 |
|---|---|---|
| 主模型超时（>30s） | timeout | 自动切 fallback_model |
| 主模型质量不达标 | review_model 评分 < G1 门槛 | 打回重试，≤2 次 |
| 主模型风格偏移 | 结构/语气检测 | 切 fallback_model |
| 主模型输出含禁用词 | G1 预扫描 | 直接打回，不重试同模型 |
| 所有模型失败 | 重试耗尽 | 标记 failed，进入人工处理 |

### 5.2 Fallback 纪律

- 单次任务主模型 → fallback 最多切 1 次
- 同一任务不重复调用同一失败模型
- fallback 输出仍必须过 G1-G6 全门
- fallback 失败不降级质检标准

## 六、模型调用日志与成本记录

### 6.1 必须记录的字段

| 字段 | 说明 |
|---|---|
| call_id | 唯一调用 ID |
| timestamp | 调用时间 |
| model_role | primary/fallback/review/rewrite |
| provider | 模型供应商 |
| model_name | 具体模型名 |
| input_tokens | 输入 token 数 |
| output_tokens | 输出 token 数 |
| cost | 本次调用成本（元） |
| latency_ms | 响应延迟 |
| success | 是否成功 |
| fail_reason | 失败原因（如有） |
| content_id | 关联的内容 ID |
| used_materials_ids | 使用的 9080 素材 ID 列表 |
| g1_result | G1 合规门结果 |

### 6.2 成本监控

- 每日汇总模型调用成本
- 单篇文案成本上限待定（M1 施工时设定）
- 免费模型调用不计成本但仍记录次数和失败率
- 成本异常（单日超阈值）自动告警

## 七、与现有架构集成

### 7.1 与 9080 安全召回的关系

```
文案加工厂智能体 → 只读调用 9080
9080 → 召回 approved 素材+合规规则+风格模板+引擎素材
文案加工厂 → 不直连 9200
```

### 7.2 与 G1-G6 门的关系

```
大模型输出 → G1 合规门（三级判定+一票否决）
            → G2 事实校验门（used_materials 覆盖度检查）
            → G3 平台结构门（四出口工艺卡匹配）
            → G4 品牌一致门（品牌调性+禁用表达）
            → G5 格式完整门（9 项结构+宪法挂靠）
            → G6 人审签发门（must_sign/signoff_approved）
```

> 注：门编号以《M1 条件施工许可 V1》第四节 V1.2.1 口径为准（G1 合规红线/G2 状态越界/G3 事实引用/G4 平台结构/G5 品牌一致/G6 格式完整），本节为 V0.1 原文保留。

### 7.3 出口约束（M1 已定）

- 文案加工厂只读调用 9080，不直连 9200
- 出口只到 candidate_review
- 不自动发布
- 不写 approved
- 不 reindex

## 八、四条硬边界（ChatGPT 摘要复核后补入）

以下 4 条为 ChatGPT V0.1 摘要复核通过后要求补入的硬边界，合入 M1 架构前必须写入。

### 8.1 硬边界一：fallback_model 也必须过 G1-G6

兜底模型不能因为是"备用"就绕过合规门、事实门、平台结构门、品牌一致门。

- fallback_model 输出与 primary_model 输出执行完全相同的 G1-G6 全门扫描
- 不得因模型切换降低任何门的判定标准
- fallback 输出同样进入 candidate_review，不得跳过任何质检环节

### 8.2 硬边界二：所有模型输出必须绑定 used_materials

没有 used_materials，不允许生成正式候选稿。

- 每份模型输出必须附带 used_materials_id 列表
- used_materials 为空时，模型不得生成正式 candidate_review 候选稿
- used_materials 为空时的唯一合法输出：**缺料报告**（标记缺失素材类型+建议召回关键词）
- 缺料报告不进入 G1-G6，不进入 candidate_review，只进入 Brief 理解层反馈循环

### 8.3 硬边界三：免费模型/低成本模型不得接触敏感数据

Kimi 等免费模型可用于受限扩写，但不得接收敏感数据。

**禁止传入免费模型的数据类型：**

- 密钥/Token/API Key
- 客户隐私（姓名/手机号/地址/消费记录）
- 未脱敏门店数据（真实店名/真实经营数据/真实客户画像）
- 未授权业务数据（未签发财务数据/未公开合作信息）

**脱敏规则：**

- 门店数据传入前必须脱敏为 store_001 / 地区-城市代号格式
- 经营数据传入前必须标注 reference_only / inferred / shadowflow_test
- 涉及真实客户信息时必须替换为虚拟占位符

### 8.4 硬边界四：模型失败必须有熔断

主模型失败、兜底失败、连续质量低、G1 连续失败、成本异常，都必须进入 stop/hold/manual_review，不得无限重试。

**熔断触发条件：**

| 触发条件 | 阈值（草案） | 熔断动作 |
|---|---|---|
| 主模型连续失败 | ≥3 次 | hold，切 fallback_model |
| 主模型+fallback 双失败 | ≥1 次 | stop，进入 manual_review |
| G1 连续 fail | ≥3 次同一内容 | stop，进入 manual_review |
| 质量评分连续低于门槛 | ≥3 次 | hold，进入 manual_review |
| 单日模型成本超阈值 | 待 M1 施工时设定 | stop，告警+人工确认 |
| 单篇文案模型调用次数 | ≥5 次 | stop，进入 manual_review |

**熔断纪律：**

- 熔断后不得自动恢复，必须人工确认后解除
- 熔断原因必须记录到调用日志
- 熔断期间同类型任务排队等待，不丢弃
